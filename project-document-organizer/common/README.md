# 通用文档规范

## 文档类型

### 通用文档
- `ARCHITECTURE.md` - 项目架构概览
- `DIRECTORY_STRUCTURE.md` - 目录结构说明
- `SETUP.md` - 项目搭建和环境配置
- `FAQ.md` - 常见问题解答
- `CHANGELOG.md` - 文档修改记录

### Java服务器项目
- `MODULES.md` - 模块划分和说明
- `API_REFERENCE.md` - API接口文档
- `DATABASE_SCHEMA.md` - 数据库结构
- `CONFIGURATION.md` - 配置说明

### Unity项目
- `SCENES.md` - 场景说明
- `SCRIPTS_ARCHITECTURE.md` - 脚本架构
- `ASSETS_ORGANIZATION.md` - 资源组织
- `BUILD_GUIDE.md` - 构建指南

### Golang服务器项目
- `PACKAGES.md` - 包结构说明
- `SERVICES.md` - 服务说明
- `PROTOCOL.md` - 协议文档
- `DEPLOYMENT.md` - 部署指南

### Cocos项目
- `SCENES_NODES.md` - 场景和节点说明
- `COMPONENTS.md` - 组件说明
- `RESOURCES.md` - 资源管理
- `PLATFORM_GUIDE.md` - 平台适配指南

## 文档输出规范

### 文档内容比对与更新策略

在生成或更新文档时，必须遵循以下内容比对策略：

#### 1. 旧文档读取
- 首先检查目标文档目录是否已存在相同的文档文件
- 如果存在，读取旧文档的完整内容

#### 2. 内容比对
- 将新生成的文档内容（排除头部信息）与旧文档内容进行比对
- 比对时忽略文档头部的HTML注释部分
- 只比对文档主体内容是否有实质性变化

#### 3. 更新决策

| 情况 | 处理方式 |
|------|----------|
| 文档不存在 | 创建新文档，设置创建时间、最后修改时间、修改人 |
| 文档存在但内容相同 | 保留旧文档，**不更新**最后修改时间和修改人 |
| 文档存在且内容不同 | 更新文档，更新最后修改时间、最后修改内容、修改人、版本号 |

#### 4. 头部信息更新规则

- **创建时间**: 文档首次创建时设置，后续永远不变
- **最后修改时间**: 仅当文档主体内容有变化时更新
- **最后修改内容**: 仅当文档主体内容有变化时更新，描述具体变更
- **修改人**: 仅当文档主体内容有变化时更新为当前执行人
- **版本**: 仅当文档主体内容有变化时递增（遵循语义化版本）

### 文档头部记录规范

所有生成的文档必须在文件开头包含以下头部信息，使用 HTML 注释格式：

```html
<!--
文档名称: [文档名称]
创建时间: [YYYY-MM-DD HH:mm:ss]
最后修改时间: [YYYY-MM-DD HH:mm:ss]
最后修改内容: [简要描述最后一次修改的内容]
修改人: [修改人姓名/团队]
版本: [v1.0.0]
项目类型: [Java/Unity/Golang/Cocos]
-->
```

#### 头部字段说明：
- **文档名称**: 文档的标题，如 "项目架构概览"
- **创建时间**: 文档首次生成的时间，格式为 YYYY-MM-DD HH:mm:ss，创建后永不修改
- **最后修改时间**: 文档最后一次修改的时间，格式同上，仅在内容有实质性变化时更新
- **最后修改内容**: 简要描述最后一次修改的内容，如 "新增数据库架构说明"，仅在内容有变化时更新
- **修改人**: 修改人的姓名或团队名称，使用激活时获取的执行人信息，仅在内容有变化时更新
- **版本**: 文档版本号，遵循语义化版本规范，仅在内容有变化时递增
- **项目类型**: 文档所属的项目类型

### CHANGELOG.md 文档修改记录规范

必须生成 `CHANGELOG.md` 文件，记录所有文档版本的修改历史。

#### CHANGELOG.md 格式：
```markdown
# 文档修改记录

## [v1.1.0] - 2026-02-25

### 新增
- 新增 `DATABASE_SCHEMA.md` 数据库结构文档
- 新增微服务架构说明到 `ARCHITECTURE.md`

### 修改
- 更新 `API_REFERENCE.md` 中的接口说明
- 修正 `MODULES.md` 中的模块依赖关系

### 修复
- 修复 `SETUP.md` 中的环境配置错误

---

## [v1.0.0] - 2026-02-20

### 新增
- 初始版本发布
- 生成所有核心文档
```

#### 修改记录规范：
1. **版本号**: 遵循语义化版本规范 (MAJOR.MINOR.PATCH)
2. **日期**: 格式为 YYYY-MM-DD
3. **分类**: 使用以下分类标签
   - `新增` - 新增的内容或文档
   - `修改` - 对现有内容的修改
   - `修复` - 修复的错误或问题
   - `移除` - 移除的内容
   - `废弃` - 即将移除的内容

### 图表规范
- 所有流程图、架构图、UML图、时序图等程序图必须使用 **Mermaid** 语法实现
- 确保图表清晰、易读，标签描述准确
- 图表应与上下文内容相关，避免冗余图表

### 强制图表要求

#### 1. UML类关系图
- **强制场景**：对于关键类关系，必须输出UML类关系图
- **适用范围**：
  - 核心业务逻辑类之间的关系
  - 数据模型类之间的关系
  - 服务接口与实现类之间的关系
- **内容要求**：包含类的主要属性和方法，清晰展示继承、实现、关联等关系

#### 2. 流程图
- **强制场景**：单个服务内的关键流程，必须输出流程图
- **适用范围**：
  - 核心业务流程
  - 重要的算法流程
  - 关键的处理流程
- **内容要求**：清晰展示流程的各个步骤、分支条件和处理逻辑

#### 3. 时序图
- **强制场景**：跨服务、跨组件、跨端的服务交互，必须输出时序图
- **适用范围**：
  - 微服务之间的调用关系
  - 前后端交互流程
  - 多系统集成场景
- **内容要求**：清晰展示各参与方之间的消息传递顺序和交互过程

### 文件格式规范
- 无特殊要求时，导出文件格式默认为 **Markdown**（.md）
- Markdown文档应遵循标准语法规范
- 代码块应指定语言类型，便于语法高亮

### 文档结构规范
- 使用统一的标题层级（#、##、###等）
- 重要内容使用粗体或列表突出显示
- 相关内容之间使用空行分隔，保持良好的可读性
