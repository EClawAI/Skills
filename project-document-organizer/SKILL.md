# 项目文档整理器

## Overview

使用此 skill 整理游戏项目的代码和可能过时的文档，生成最新的、完整的技术文档，方便 AI 后续通过文档和代码快速定位问题。支持多种游戏开发技术栈（Java、Unity、Golang、Cocos）。

**Announce at start:** "我正在使用 project-document-organizer skill 来帮你整理项目文档！"

## 配置说明

### 可配置的目录

1. **旧文档目录** - 存放项目旧文档的位置
2. **项目目录** - 项目源代码所在的位置
3. **新文档目录** - 生成的新文档输出位置

### 文档模板

技能使用以下模板生成文档：

- `templates/architecture_template.md` - 架构文档模板
- `templates/module_template.md` - 模块概览文档模板
- `templates/detailed_design_template.md` - 详细功能设计文档模板
- `templates/changelog_template.md` - 变更日志模板
- `templates/directory_structure_template.md` - 目录结构说明模板

### 配置方式

- 如果已预先配置好目录，skill 会自动使用配置的路径
- 如果没有配置，激活 skill 时会提示用户确认这三个目录

### 配置文件（可选）

可以在 skill 目录下创建 `config/config.json` 配置文件，路径为：`.trae/skills/project-document-organizer/config/config.json`，格式如下：

```json
{
  "oldDocsDir": "/path/to/old/docs",
  "projectDir": "/path/to/project/source",
  "newDocsDir": "/path/to/new/docs"
}
```

### 具体激活步骤

1. **[1] 执行人信息**
   请问本次文档整理的执行人是谁？
   • 匿名用户（默认）
   • Others: 输入执行人姓名

2. **[2] 旧文档目录**
   请确认旧文档目录（如果没有可以留空）
   • 留空
   • Others: 输入旧文档目录路径

3. **[3] 项目目录**
   请确认项目目录
   • Others: 输入项目目录路径

4. **[4] 新文档目录**
   请确认新文档目录
   • Others: 输入新文档目录路径

5. **[5] 文档规模**
   请选择文档生成规模
   • 简单：只生成基础组件和架构结构性文档及主要模块概述，用于快速了解项目
     - 详细规范请参考：`simple/README.md`
   • 一般：除简单部分外，额外生成核心模块的详细文档
     - 详细规范请参考：`medium/README.md`
   • 详细：尽可能详细说明各个组件、架构、模块及内部实现
     - 详细规范请参考：`detailed/README.md`

6. **[6] 其他补充**
   是否有更多的补充信息需要提供？（可选）
   • 否（默认）
   • Others: 输入补充信息

7. 确认所有信息后开始执行



### 文档生成流程

**重要提示：不管是哪种文档规模，都需要遵循通用文档规范的内容。详细规范请参考：`common/README.md`**

#### 通用流程
0. **和用户确认各种信息**
   - 确认执行人、目录路径、文档规模等信息
   - 确保所有必要信息都已提供

1. **分析项目结构**
   - 识别项目类型和技术栈
   - 分析目录结构和模块划分
   - 自动检测所有子项目和功能模块

2. **创建文档目录结构**
   - 创建根目录文档（ARCHITECTURE.md, DIRECTORY_STRUCTURE.md等）
   - 根据文档规模创建相应的目录结构

3. **递归分析文件内容**
   - 按系统或功能对文件进行归类
   - 生成临时归类文档
   - 不同文档规模对应不同的归类详细程度

4. **和用户确认临时归类文档**
   - 展示生成的临时归类文档
   - 确认细分程度是否满足需求
   - 根据用户反馈调整归类

5. **生成整体架构文档**
   - 生成项目整体架构图
   - 创建架构概览文档

6. **检查文档完整性**
   - 检查是否完成了所有必要的文档
   - 如果发现缺失则补全
   - 确保所有必要的文档都已生成

7. **整合文档**
   - 生成文档之间的交叉引用
   - 确保文档结构清晰、层次分明

8. **质量检查**
   - 检查文档内容的完整性和准确性
   - 确保图表的正确性和清晰度
   - 检查文档格式和规范是否符合要求

9. **生成CHANGELOG.md**
   - 记录文档修改历史
   - 包含版本变更信息

10. **删除临时文件**
    - 删除中间过程产生的临时文件，如：临时归类文档

#### 简单文档规模流程
1. **分析项目结构**（简化版）
   - 识别项目类型和技术栈
   - 分析顶层目录结构
   - 识别主要模块

2. **创建基础文档结构**
   - 仅创建根目录文档
   - 不创建子模块目录

3. **生成基础文档**
   - 生成 ARCHITECTURE.md - 项目架构概览（简化版）
   - 生成 DIRECTORY_STRUCTURE.md - 目录结构说明
   - 生成 SETUP.md - 项目搭建和环境配置
   - 生成 FAQ.md - 常见问题解答
   - 生成 CHANGELOG.md - 文档修改记录

4. **质量检查**
   - 确保基础文档完整
   - 检查文档格式规范

#### 一般文档规模流程
1. **分析项目结构**
   - 识别项目类型和技术栈
   - 分析目录结构和模块划分
   - 识别核心模块和子项目

2. **创建文档目录结构**
   - 创建根目录文档
   - 为核心模块创建子目录

3. **生成文档**
   - 根目录文档：
     - ARCHITECTURE.md - 项目架构概览
     - DIRECTORY_STRUCTURE.md - 目录结构说明
     - SETUP.md - 项目搭建和环境配置
     - FAQ.md - 常见问题解答
     - CHANGELOG.md - 文档修改记录
   - 核心模块文档：
     - MODULES.md - 模块划分和说明
     - API_REFERENCE.md - API接口文档（如适用）
     - DATABASE_SCHEMA.md - 数据库结构（如适用）
     - CONFIGURATION.md - 配置说明

4. **质量检查**
   - 确保所有核心模块都有对应的文档
   - 检查文档内容的完整性

#### 详细文档规模流程
1. **分析项目结构**
   - 递归分析项目目录结构
   - 识别所有子项目和功能模块
   - 分析模块之间的依赖关系

2. **创建详细文档目录结构**
   - 创建根目录文档
   - 为每个子项目创建独立的文件夹目录
   - 为每个功能模块创建相应的文件夹目录

3. **生成整体架构文档和模块总览**
   - 生成项目整体架构图
   - 创建模块总览文档，包含所有模块的概览

4. **为每个子项目生成详细架构文档**
   - 为每个子项目生成独立的架构图
   - 创建子项目级别的架构文档

5. **深度优先生成详细文档**
   - 逐个对每个子项目进行处理
   - 以临时归类文档中的每个系统、功能或模块为基础
   - 生成详细功能设计文档（详细规范请参考：`detailed/README.md`）
   - 生成相关UML类关系图、流程图、时序图

6. **检查文档完整性**
   - 检查是否完成了每个子项目的每个归类
   - 如果发现缺失则补全
   - 确保所有模块都有对应的详细文档

7. **整合文档**
   - 在总模块文档中添加各个子文档的引用
   - 生成文档之间的交叉引用
   - 确保文档结构清晰、层次分明

8. **质量检查**
   - 检查文档内容的完整性和准确性
   - 确保图表的正确性和清晰度
   - 检查文档格式和规范是否符合要求


### 执行人信息获取

## 功能说明

### 支持的项目类型

1. **Java游戏服务器项目** - 基于Java的游戏后端服务
2. **Unity客户端项目** - Unity游戏开发项目
3. **Golang服务器项目** - 基于Go的游戏后端服务
4. **Cocos客户端项目** - Cocos游戏开发项目

### 多子项目和多模块支持

对于包含多个子项目（如gameServer、client、chatServer、gateServer等）和多个子模块（如任务模块、联盟模块、存储模块等）的复杂项目，文档生成将采用以下结构：

#### 1. 整体架构文档
- 生成项目整体架构图，展示各个子项目之间的关系
- 提供项目总览和子项目列表

#### 2. 子项目文档
- 为每个子项目生成独立的架构图
- 为每个子项目创建独立的文件夹目录
- 在子项目目录中生成该子项目的详细文档
- 自动识别子项目中的功能模块，无需用户二次说明

#### 3. 模块文档
- 为子项目中的每个功能模块生成详细文档
- 以树状结构组织模块文档，严格按照目录层级结构生成
- 在总模块文档中包含每个子模块的说明介绍
- 在总文档中添加各个子文档的引用和说明
- 对于详细文档模式，深入分析模块内部系统，生成完整的内部实现文档

#### 4. 文档目录结构示例

## 使用步骤

### 1. 项目结构分析
首先对项目进行全面的结构分析：
- 识别项目类型（Java/Unity/Golang/Cocos）
- 分析代码目录结构
- 查找现有的旧文档
- 识别关键模块和组件

### 2. 代码分析
深入分析代码，提取以下信息：
- 主要模块和它们的职责
- 核心类/结构体和接口
- 关键算法和数据结构
- 配置文件和环境变量
- 依赖关系
- 数据库结构（如适用）
- API接口（如适用）

### 3. 文档对比与整合
- 对比现有旧文档与实际代码
- 识别文档中的过时信息
- 保留仍有效的文档内容
- 补充缺失的文档

### 4. 生成新文档
根据分析结果，生成相应类型的文档。详细文档类型和规范请参考：`common/README.md`

## 何时使用此技能

### 1. 项目交接场景
- **接手新项目时** - 从其他团队或开发者手中接过项目，需要快速了解项目全貌
- **团队成员离职交接** - 确保项目文档完整，便于后续维护
- **跨团队协作接手** - 从其他部门或外包团队接手项目

### 2. 文档管理场景
- **文档明显过时时** - 发现现有文档与代码不一致，需要更新
- **缺少关键文档时** - 项目没有完整的技术文档，需要从零开始生成
- **文档分散混乱时** - 文档分布在多个位置，需要整合梳理
- **策划文档与代码脱节** - 游戏策划文档与实际实现存在差异

### 3. 开发工作场景
- **需要快速了解项目结构时** - 新功能开发前需要熟悉项目架构
- **准备进行大规模重构前** - 重构前建立完整的基线文档
- **Bug定位困难时** - 通过文档快速定位问题区域
- **性能优化前** - 了解系统架构，找出优化点

### 4. 团队协作场景
- **新人入职培训前** - 为新员工准备完整的项目文档
- **技术分享准备** - 需要系统性地介绍项目时
- **代码审查辅助** - 有完整文档便于理解代码上下文
- **多项目并行管理** - 同时管理多个项目，需要统一的文档标准

### 5. 特殊场景
- **项目复活时** - 重启长期搁置的项目
- **技术审计前** - 为外部审计准备完整的技术文档
- **开源准备** - 准备将项目开源，需要完整的文档
- **知识产权梳理** - 需要梳理项目技术资产时

### 6. 定期维护
- **每季度/半年定期更新** - 保持文档与代码同步
- **重大版本发布后** - 记录新版本的架构变更
- **技术栈升级后** - 更新相关技术文档

## 最佳实践

1. 从项目根目录开始，逐层深入分析
2. 优先分析核心业务逻辑
3. 注重代码和文档的一致性
4. 保持文档简洁但信息完整
5. 使用清晰的结构和一致的格式

## 错误处理

- **目录不存在**: 提示用户检查路径是否正确
- **项目类型识别失败**: 提供项目类型选项供用户选择
- **代码分析超时**: 提供跳过部分模块的选项
- **文档生成失败**: 给出详细的错误信息和重试建议
- **权限不足**: 提示用户检查目录读写权限
