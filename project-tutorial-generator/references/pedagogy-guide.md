# 教学设计原则

本文档定义了 project-tutorial-generator 的教学设计原则，包括用户级别影响矩阵、质量检查标准和教学效果评估标准。

---

## 1. 用户级别影响矩阵

用户级别（Phase 1 选择）决定教程内容的多个维度。以下矩阵指导各维度在不同级别下的表现。

### 1.1 内容深度维度

| 维度 | 级别 1-2（入门向） | 级别 3（中间） | 级别 4-5（进阶向） |
|------|-------------------|---------------|-------------------|
| **基础概念解释** | 详细。每引入一个技术概念，用 1-2 段文字解释其含义、为什么需要它 | 简略。用一句话说明概念，附链接供深入阅读 | 跳过。假设读者已知 |
| **代码片段量** | 少而精。只贴最核心的 3-5 行，每行配注释 | 适中。贴关键实现，块级注释 | 多而全。包含实现细节、边界处理、配置代码 |
| **架构决策讨论** | 简化。给出结论和一句话理由 | 展开。列出 2-3 个备选方案，解释选择理由 | 深入。完整的方案对比表，含性能、可维护性、团队因素 |
| **Mermaid 图表** | 简化概览图。模块关系、简单流程 | 标准图。含关键交互和数据流 | 详细交互图。时序图、状态机、完整类图 |
| **"为什么不用 X"讨论** | 不涉及 | 偶尔提及 | 每个决策点都讨论替代方案 |
| **性能/运维考量** | 不涉及 | 提及关键点 | 展开讨论。含基准数据、监控策略 |
| **代码设计模式解读** | 不涉及 | 命名并简述 | 详细解析模式意图、适用场景 |

### 1.2 叙事风格

| 级别 | 叙事方式 |
|------|---------|
| 1-2 | **手把手引导**。"首先我们需要...因为..."。像导师带新人 |
| 3 | **同行讨论**。"这里选择了 X 方案，因为..."。像同事间技术分享 |
| 4-5 | **架构复盘**。"核心权衡是 A vs B，我们选了 A 基于以下考量..."。像技术评审 |

### 1.3 每章预期篇幅

| 级别 | 每章大约字数 | 代码占比 |
|------|-------------|---------|
| 1-2 | 3000-5000 字 | ~20% |
| 3 | 2000-4000 字 | ~30% |
| 4-5 | 2000-3000 字 | ~40% |

## 2. 章节自检清单

每章生成后，必须通过以下检查项。

### 2.1 结构完整性

- [ ] 包含"问题引入"部分，清晰描述此迭代步骤面临的问题
- [ ] 包含"设计决策"部分，至少提到选择的方案和一个理由
- [ ] 包含"核心实现"部分，有至少一个代码片段配注释
- [ ] 包含"演进复盘"部分，说明此步带来的改进
- [ ] 包含"扩展思考"部分，至少提供一个思考方向

### 2.2 代码可读性

- [ ] 代码片段有语言标注（如 ```java, ```go）
- [ ] 关键行有注释解读
- [ ] 片段自包含——不阅读源码也能理解
- [ ] 跨版本对比使用 Before/After 格式，标注变化点

### 2.3 图表完备性

- [ ] 章节包含至少一个图表（Mermaid 或绘图描述）
- [ ] Mermaid 代码块语法正确（可被标准渲染器解析）
- [ ] 绘图描述包含：类型、标题、布局、元素、连接、标注
- [ ] 图表辅助理解正文内容，不是孤立存在

### 2.4 用户级别对齐

- [ ] 概念解释详细度符合所选级别
- [ ] 代码量符合级别预期
- [ ] 架构讨论深度符合级别预期

---

## 3. 章节间一致性检查

所有章节生成完毕后，执行以下跨章检查。

### 3.1 术语一致性

- 同一技术概念在所有章节中使用相同术语
- 示例：如果第2章用"网关服务"，第5章不能突然改叫"Gate 服务"
- 规则：以最早引入该概念的章节中使用的术语为准

### 3.2 引用完整性

- 章节间交叉引用指向正确的章节编号
- 不存在引用了不存在的章节
- 引用格式统一：如"参见第3章"或"如第3章所述"

### 3.3 渐进复杂度

- 后面的章节不能假设读者了解本章才会介绍的概念
- 如果必须前向引用，需加简要说明："我们将在第N章详细讨论，这里只需了解..."
- 知识依赖链应该是单向的（前→后）

### 3.4 无重复内容

- 同一段解释不在多个章节中出现
- 如果多章需要相同背景知识，后续章节用简短引用代替完整重复

## 4. 教学效果评估标准

### 4.1 概念密度控制

- 每章引入的**主要新概念**不超过 2-3 个
- "主要概念"定义：需要读者花时间理解的技术概念（不计入已知概念的简单应用）
- 如果一章需要引入 4+ 个新概念，建议拆分

### 4.2 实例配合原则

- 每个抽象概念必须有具体实例
- 实例来自项目真实代码，不使用伪代码或假想示例
- 代码实例紧跟概念解释，不要隔太远

### 4.3 Before/After 清晰性

- 每章应有明确的"改变前"和"改变后"对比
- 对比维度：代码结构、架构图、能力边界
- 让读者清楚看到"这一步带来了什么改进"

### 4.4 可验证学习目标

- 每章开头隐含或显式声明学习目标
- 章末的"扩展思考"应该让读者能自检是否达成目标
- 好的学习目标示例："理解为什么要用 gRPC Stream 而不是 REST"
- 差的学习目标示例："了解 gRPC"

### 4.5 叙事连贯性

- 教程整体讲述一个连贯的故事：从简单到复杂
- 每章开头简短回顾上一章的成果
- 每章结尾预告下一章要解决的问题
- 读者应该能回答："为什么这一步是现在做，而不是更早/更晚？"
