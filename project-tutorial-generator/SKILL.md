# Project Tutorial Generator

**Announce at start:** "我正在使用 project-tutorial-generator skill 来为你的项目生成迭代式教程！"

## Overview

接收一个项目路径，从四个信息源（git history、openspec proposals、已有设计文档、代码结构）提取教学素材，将项目的演进历史重构为一条清晰的教学路径，生成可独立阅读的 Markdown 系列教程。

与 `project-document-organizer` 的区别：
- **document-organizer**：最终态快照——"这个系统是什么"
- **tutorial-generator**：演进过程重建——"这个系统是怎么一步步长成这样的"

## Configuration

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| project_path | string | (required) | 项目根目录路径 |
| output_dir | string | (交互确认) | 教程输出目录 |
| audience_level | int | (交互选择) | 目标用户级别（1-5，动态生成） |
| generation_mode | enum | batch | 生成模式：batch（一次性）/ chapter-by-chapter（逐章确认） |

## Workflow — 六阶段流水线

```
Phase 0 → Phase 1 → Phase 2 → Phase 3 → Phase 4 → Phase 5
初始化     用户定级   项目分析   大纲确认   内容生成   质量检查
```

---

## Phase 0: 初始化

### Step 0.1: 接收项目路径

从用户输入中获取项目路径。如果用户未提供，使用 AskQuestion 询问：

```
请提供要生成教程的项目路径（绝对路径）：
```

验证路径存在且是有效的项目目录（包含代码文件）。

### Step 0.2: 确认教程输出目录

询问用户教程的输出位置：

```
请确认教程输出目录：
• 默认: <project_path>/tutorial/
• Others: 输入自定义路径
```

创建输出目录（如不存在）。

### Step 0.3: 快速扫描项目

执行快速扫描，收集项目基本信息：

1. **技术栈检测**：扫描构建文件（pom.xml, package.json, go.mod, Cargo.toml 等）确定语言和框架
2. **规模评估**：统计源代码文件数、目录层级深度
3. **openspec 检测**：检查是否存在 `openspec/` 目录及其中的 changes
4. **文档检测**：扫描 `docs/`, `doc/`, 根目录 `.md` 文件
5. **git 检测**：检查是否有 git 历史，统计 commit 数量

向用户展示扫描结果：

```
项目扫描结果：
─────────────
路径: /path/to/project
技术栈: Java / Spring Boot / gRPC / Redis
规模: ~150 个源文件，12 个模块
Git: 有，共 45 个 commit
OpenSpec: 有，25 个归档变更
设计文档: 发现 15 个 .md 文件在 docs/
```

### Step 0.4: 创建 context.md

在输出目录创建 `context.md`，记录初始信息：

```markdown
# Tutorial Generator Context

## 项目信息
- 路径: <project_path>
- 技术栈: <detected tech stack>
- 规模: <file count, module count>
- Git: <有/无, commit count>
- OpenSpec: <有/无, change count>
- 设计文档: <发现的文档数量和位置>

## 目标用户
(Phase 1 完成后填入)

## 演进节点与决策
(Phase 2 完成后填入)

## 确认的大纲
(Phase 3 完成后填入)
```

**完成 Phase 0 后，进入 Phase 1。**

---

## Phase 1: 目标用户定级

详见 Step 内容。参考 `references/pedagogy-guide.md` 中的用户级别影响矩阵。

### Step 1.1: 动态生成用户级别

基于 Phase 0 检测到的技术栈和项目领域，动态生成 3-5 档目标用户级别。**不使用**通用的"初级/中级/高级"，而是生成与项目技术栈相关的具体级别。

生成规则：
- 第 1 档：该技术栈的入门者（需要解释基础概念）
- 第 2 档：有基础经验但缺少项目核心领域经验
- 第 3 档：有相关领域经验，但未接触过此类项目
- 第 4 档：领域专家，关注架构决策和实现细节
- 第 5 档：项目接手者/维护者，关注项目特定设计和扩展点

### Step 1.2: 展示给用户选择

以交互式列表展示，每个级别附带说明（教程会强调什么、跳过什么）。

### Step 1.3: 记录到 context.md

将用户选择和对应的深度策略记录到 `context.md` 的"目标用户"部分。

**完成 Phase 1 后，进入 Phase 2。**

---

## Phase 2: 项目深度分析

详细分析策略参考 `references/analysis-guide.md`。

### Step 2.1: 分析四个信息源

按照 `references/analysis-guide.md` 中定义的策略，依次分析：
1. **git history** — 还原演进脉络
2. **openspec proposals** — 提取设计决策（如有）
3. **已有设计文档** — 扫描并索引
4. **代码结构** — 模块、依赖、关键类

### Step 2.2: 识别关键演进节点

根据 `references/analysis-guide.md` 中的双模式逻辑：

**有 openspec（富信息模式）：**
- 每个归档的 change 作为一个演进节点
- 从 proposal.md 提取迭代动机
- 从 design.md 提取设计决策
- 从 tasks.md 了解实现粒度

**无 openspec（交互探查模式）：**
- 从 git history 识别重大变化点（功能提交、重构、依赖变更）
- 对每个节点逐一询问用户迭代原因
- 用户回答 → 记录来源为"用户回答"
- 用户跳过/不清楚 → AI 推理，记录来源为"AI 推理"

### Step 2.3: 记录到 context.md

将所有演进节点、迭代原因、设计决策记录到 `context.md`，每条标注信息来源。

**完成 Phase 2 后，进入 Phase 3。**

---

## Phase 3: 大纲生成与确认

大纲模板参考 `templates/course-outline.md`。

### Step 3.1: 生成预览大纲

基于 Phase 2 的分析结果，将演进节点重组为教学路径。大纲预览包含四层信息：

```
第N章: [章节名称]
简介: [章节内容简介 — 解决什么问题、学到什么]

  N.1 [小节名称]
      [小节内容简介]
  N.2 [小节名称]
      [小节内容简介]
```

教学路径可能不同于 git 真实提交顺序——按最佳学习序列重排。

### Step 3.2: 交互修订循环

将预览大纲展示给用户，接受任意修改。支持六种操作：
1. **重排** — 调整章节顺序
2. **拆分** — 一章拆多章
3. **合并** — 多章并一章
4. **增删** — 添加或移除章节
5. **调深度** — 某章讲深或讲浅
6. **调视角** — 改变叙事角度

**不限轮次**，用户明确确认后才进入下一阶段。

### Step 3.3: 选择生成模式

大纲确认后，让用户选择：

```
请选择教程生成模式：
[1] 一次性生成 — 所有章节连续生成，无需中间确认
[2] 逐章确认 — 每生成一章后暂停，可选择：继续/重写/调深/调浅
```

### Step 3.4: 记录最终大纲

将确认的大纲和生成模式记录到 `context.md`。

**完成 Phase 3 后，进入 Phase 4。**

---

## Phase 4: 内容生成

章节模板参考 `templates/chapter-template.md`，图表模式参考 `templates/diagram-patterns.md`。

### Step 4.1: 生成第 0 章——课程概述

文件名: `00-课程概述.md`

内容：
1. 项目全景架构图（Mermaid 或绘图描述）
2. 技术栈说明（含版本信息）
3. 教程路线图——所有章节及其关系
4. 前置知识要求（基于用户级别）

### Step 4.2: 逐章生成

按确认的大纲顺序，逐章生成内容。每章遵循五段式结构：

1. **问题引入** — 此演进节点面临的问题/需求
2. **设计决策** — 有哪些方案、为何选了当前方案
3. **核心实现** — 关键代码片段 + 注释解读（自包含）
4. **演进复盘** — 改了什么、改进了什么、留了什么坑
5. **扩展思考** — 替代方案、业界做法、读者可尝试什么

文件命名: `01-章节标题.md`, `02-章节标题.md`, ...

**代码引用规范**（详见 `templates/chapter-template.md`）：
- 贴关键片段，配注释解读
- 跨版本对比用 Before/After 格式
- 确保只看教程就能理解，不依赖源码

**图表策略**（详见 `templates/diagram-patterns.md`）：
- Mermaid 可表达的 → 直接生成 Mermaid 代码块
- Mermaid 不可表达的 → 生成结构化绘图描述
- 每章至少包含一个架构/流程相关图表

### Step 4.3: 生成模式处理

**一次性模式**: 连续生成所有章节。

**逐章确认模式**: 每章生成后暂停，展示给用户，等待指令：
- `继续` → 生成下一章
- `重写` → 重新生成本章
- `调深` → 增加细节重新生成
- `调浅` → 简化内容重新生成

**完成 Phase 4 后，进入 Phase 5。**

---

## Phase 5: 质量检查

质量标准参考 `references/pedagogy-guide.md`。

### Step 5.1: 章节自检

对每章执行自检清单：
- [ ] 五段式结构完整（问题引入/设计决策/核心实现/演进复盘/扩展思考）
- [ ] 代码片段有注释解读
- [ ] 至少包含一个图表（Mermaid 或绘图描述）
- [ ] 内容深度符合目标用户级别

发现问题自动修复。

### Step 5.2: 章节间一致性检查

- 术语一致性（同一概念全篇用同一名称）
- 引用完整性（章节交叉引用指向正确）
- 渐进复杂度（不提前引用后面才介绍的概念）
- 无重复内容

### Step 5.3: 输出完成摘要

```
教程生成完成！
──────────────
输出目录: <output_dir>
章节数量: <N> 章
预估阅读时间: <M> 分钟

章节列表:
  00-课程概述.md — 项目全景与教程路线图
  01-xxx.md — ...
  02-xxx.md — ...
  ...

context.md 已保留在输出目录中。
```

---

## Error Handling

- **项目路径不存在**: 提示用户检查路径
- **无 git 历史且无 openspec**: 仅基于代码结构和设计文档生成，警告教程质量可能受限
- **无任何信息源**: 报错退出，提示至少需要代码文件
- **用户中断**: 已完成的章节保留，context.md 保存当前进度，下次可续写
