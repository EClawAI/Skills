# 课程大纲模板

本模板定义了大纲预览的展示格式。Skill 在 Phase 3 中使用此模板生成预览大纲。

---

## 大纲预览格式

每个章节必须包含四层信息：章节名称、章节内容简介、小节名称、小节内容简介。

```markdown
# 教程大纲预览

## 第0章: 课程概述
简介: 项目全景、技术栈介绍、教程路线图，帮助读者建立整体认知。

  0.1 项目全景架构图
      展示项目最终形态的整体架构，让读者知道"我们要构建什么"。
  0.2 技术栈说明
      列出项目使用的语言、框架、中间件及其版本。
  0.3 教程路线图
      以可视化方式展示各章节的学习路径和依赖关系。
  0.4 前置知识要求
      基于目标用户级别，列出阅读本教程需要的前置知识。

---

## 第1章: [章节名称]
简介: [这章要解决什么问题、读者将学到什么——1-2 句话]

  1.1 [小节名称]
      [这一小节具体讲什么——1 句话]
  1.2 [小节名称]
      [这一小节具体讲什么——1 句话]
  1.3 [小节名称]
      [这一小节具体讲什么——1 句话]

## 第2章: [章节名称]
简介: [这章要解决什么问题、读者将学到什么]

  2.1 [小节名称]
      [这一小节具体讲什么]
  2.2 [小节名称]
      [这一小节具体讲什么]

...（继续）
```

## 大纲生成指导

### 章节名称规范

- 用问题或动作命名，而非抽象概念
- 好的命名：`从零搭建 Gate-Game 通信链路`、`为什么需要拆分出 CenterService`
- 差的命名：`gRPC 通信`、`服务配置`

### 章节简介规范

- 必须说明"解决什么问题"和"学到什么"
- 好的简介：`当服务数量增多，每个服务单独管理配置变得不可维护。本章讲解如何引入 CenterService 统一配置管理，以及服务注册发现的设计。`
- 差的简介：`讲解 CenterService 的实现。`

### 小节组织原则

- 每章 2-5 个小节为宜
- 小节顺序应该符合认知递进
- 首个小节通常是"为什么要做这件事"
- 末个小节通常是"总结与思考"或具体的实现细节

### 教学路径重排

大纲的章节顺序按最佳学习序列排列，可能不同于 git 真实提交顺序。重排原则：

1. **依赖关系优先**：被后续章节依赖的概念必须先讲
2. **从核心到外围**：先讲核心功能，再讲辅助系统（监控、安全、部署）
3. **从简单到复杂**：先建立最小可用系统，再逐步增强
4. **聚合相关变更**：将同一主题的多次迭代合并到一章

## 交互修订指引

当用户反馈修改时，按以下方式处理：

| 用户操作 | 处理方式 |
|---------|---------|
| **重排** "A 章放到 B 章后面" | 调整章节顺序，更新编号，检查是否产生依赖冲突 |
| **拆分** "第3章太大，拆开" | 将一章拆为两章，重新分配小节，各自补充简介 |
| **合并** "这两章合一起" | 将两章合并，整合小节，更新简介 |
| **增加** "加一章讲 XXX" | 在合适位置插入新章节，生成小节和简介 |
| **删除** "不用讲 XXX" | 移除章节，检查是否有其他章引用它 |
| **调深度** "这章讲深一点" | 增加小节数或扩展简介中的覆盖范围 |
| **调视角** "从架构角度讲，不要从代码角度" | 调整章节简介和小节命名以反映新视角 |

每次修改后重新展示完整大纲，供用户继续审阅。
