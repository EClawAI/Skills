## Context

当前工作区 (`~/Work/Lzw/Skills`) 已有 `project-document-organizer` skill，用于生成项目最终态的技术参考文档。新的 `project-tutorial-generator` skill 与其完全独立，但定位互补：

- **document-organizer**：最终态快照——"这个系统是什么"
- **tutorial-generator**：演进过程重建——"这个系统是怎么一步步长成这样的"

Skill 文件结构遵循已有惯例（SKILL.md + templates/ + references/），安装到 `~/.agents/skills/project-tutorial-generator/`。

参考项目 GateDemo (`~/Work/Lzw/GateDemo/`) 拥有完整的 git history（20+ commits）、25+ 个归档 openspec changes、大量中文设计文档，是理想的验证素材。

## Goals / Non-Goals

**Goals:**

- 创建一个通用的 skill，接收任意项目路径，生成按迭代步骤组织的教程文档
- 支持有 openspec 和无 openspec 两种项目，行为自动分化
- 生成的教程纯阅读就有价值，不依赖源码
- 交互式大纲确认，用户完全掌控教程结构
- 尽可能多用 Mermaid 图表和绘图描述辅助说明
- SKILL.md 控制在 500 行以内

**Non-Goals:**

- 不替代 document-organizer，不生成 API 参考文档或模块说明
- 不生成可运行的代码项目或脚手架
- 不支持自动发布到博客/文档站点
- 不做视频/音频等多媒体教程
- 不做跨项目的教程合集

## Decisions

### D1: 工作流设计——六阶段流水线

**决策**: 采用六阶段线性流水线，Phase 0-5 顺序执行。

```
Phase 0: 初始化
  ① 接收项目路径
  ② 确认教程输出目录
  ③ 快速扫描项目基本信息
  ④ 在输出目录创建 context.md

Phase 1: 目标用户定级
  ① 基于技术栈动态生成 3-5 档目标用户
  ② 用户选择
  ③ 记录到 context.md

Phase 2: 项目深度分析
  ① 分析四个信息源（git / openspec / docs / code）
  ② 识别关键演进节点
  ③ 有 openspec: proposal 直取迭代原因/决策
  ④ 无 openspec: 逐节点交互询问用户，跳过则 AI 推理
  ⑤ 所有信息记录到 context.md

Phase 3: 大纲生成与确认
  ① 生成预览大纲（章节名+简介、小节名+简介）
  ② 不限轮次交互修订，用户确认为准
  ③ 确认前让用户选择生成模式（一次性/逐章确认）
  ④ 最终大纲记录到 context.md

Phase 4: 内容生成
  ① 生成第0章（课程概述：全景图+技术栈+路线图）
  ② 按大纲逐章生成
  ③ 一次性模式：全部生成
  ④ 逐章确认模式：每章后暂停，支持重写/调深/调浅
  ⑤ 每章结构：问题引入→设计决策→核心实现→演进复盘→扩展思考

Phase 5: 质量检查
  ① 章节自检清单
  ② 章节间一致性检查
  ③ 输出完成摘要
```

**替代方案**: 考虑过并行分析+生成的方案，但教程有严格的前后依赖（后章引用前章概念），线性流水线更可靠。

### D2: 有/无 openspec 的双模式分化

**决策**: Skill 在 Phase 2 检测项目是否有 openspec 目录，自动选择工作模式。

| 维度 | 富信息模式（有 openspec） | 交互探查模式（无 openspec） |
|------|-------------------------|---------------------------|
| 迭代粒度 | openspec tasks 驱动 | AI 分析 git diff + code 推理 |
| 迭代原因 | proposal.md 直取 | 逐节点询问用户，跳过则 AI 推理 |
| 决策思考 | design.md 直取 | 逐节点询问用户，跳过则 AI 推理 |
| 教程质量 | 高（有明确文档支撑） | 中（部分依赖推断） |

**替代方案**: 考虑过统一用 AI 推理（不区分模式），但有 openspec 时信息质量远高于推理结果，不利用是浪费。

### D3: 目标用户级别——动态生成而非固定档次

**决策**: 不使用"初级/中级/高级"通用分层，而是基于项目技术栈动态生成 3-5 档。

示例（Java/Spring Boot/gRPC/Redis 项目）：
1. Java 初学者 → 详细解释基础概念
2. Java 开发者（无微服务经验）→ 重点讲服务拆分动机
3. 有微服务经验的后端开发者 → 聚焦游戏网关特殊性
4. 游戏后端开发者 → 直入架构决策对比
5. 项目维护者/接手者 → 侧重项目特定设计和坑点

用户选择后影响：概念解释详细度、代码片段量、架构讨论深度、图表复杂度。

**替代方案**: 固定三档（入门/进阶/专家），更简单但无法适配不同技术栈项目的特殊性。

### D4: 大纲预览格式

**决策**: 预览大纲包含四层信息——章节名称、章节内容简介、小节名称、小节内容简介。

```markdown
## 第1章: [章节名称]
简介: [这章要解决什么问题、学到什么]

  1.1 [小节名称]
      [这一小节具体讲什么]
  1.2 [小节名称]
      [这一小节具体讲什么]
```

用户可做的调整：重排顺序、拆分、合并、增删章节、调深度、调视角。不限交互轮次。

### D5: 图表策略——Mermaid 优先 + 绘图描述兜底

**决策**: 尽可能多用图表说明。

- 能用 Mermaid 的（流程图、时序图、类图、状态图、gitgraph）→ 直接生成 Mermaid 代码块
- 不能用 Mermaid 表达的（部署拓扑、复杂 UI 交互、性能对比图等）→ 生成结构化绘图描述

绘图描述格式规范：
```
> **[图表类型] 图表标题**
> 布局：...
> 元素：...
> 连接：...
> 标注：...
```

描述足够精确，用户可据此手绘或交给绘图 AI。

### D6: context.md 永久保留

**决策**: context.md 存放在教程输出目录，教程完成后不清理。

内容包括：项目信息、目标用户级别、演进节点与决策（含信息来源标注）、确认的最终大纲。

作用：教程元数据、可恢复（中断后继续）、可审计（用户可纠正信息）。

### D7: 章节内容结构

**决策**: 每章采用固定五段式结构：

1. **问题引入** — 在这个演进节点，面临什么问题/需求
2. **设计决策** — 有哪些方案、为什么选了当前方案
3. **核心实现** — 关键代码片段 + 注释解读（自包含，不依赖源码）
4. **演进复盘** — 这步改了什么、带来什么改进、留了什么坑
5. **扩展思考** — 还能怎么做、业界其他方案、读者可以尝试什么

### D8: 代码引用策略

**决策**: 贴关键片段 + 注释解读。不要求读者打开源码。

- 选取最能说明设计意图的代码片段
- 每个片段配行内注释或段落解读
- 跨版本对比时，用"Before/After"格式展示
- 不贴完整文件，只贴核心逻辑

### D9: 输出目录结构

**决策**: 所有产物平铺在用户指定的输出目录下。

```
<输出目录>/
├── context.md          # 元数据（永久保留）
├── 00-课程概述.md       # 全景图 + 技术栈 + 路线图
├── 01-xxx.md           # 第1章
├── 02-xxx.md           # 第2章
├── ...
└── 0N-xxx.md           # 第N章
```

### D10: Skill 文件组织

**决策**: 遵循 project-document-organizer 的组织惯例。

```
project-tutorial-generator/
├── SKILL.md                    # 主入口（<500行）
├── templates/
│   ├── course-outline.md       # 大纲模板
│   ├── chapter-template.md     # 章节模板
│   └── diagram-patterns.md     # Mermaid 图表模式库
└── references/
    ├── analysis-guide.md       # 项目分析指南（四个信息源）
    └── pedagogy-guide.md       # 教学设计原则
```

## Risks / Trade-offs

**[R1] 无 openspec 项目的教程质量不可控** → 通过交互式询问补充信息；AI 推理部分标注来源为"推断"，让读者知晓

**[R2] 大型项目分析耗时长** → Phase 2 先做快速扫描展示基本信息，深度分析按需进行；大纲预览让用户提前砍掉不需要的章节

**[R3] 用户在大纲确认环节陷入死循环** → Skill 在每轮修订后主动提示"确认后即可开始生成"；不强制最大轮次但引导推进

**[R4] 教程章节数失控（项目太大）** → 在 Phase 1 用户定级后，根据级别给出建议章节数范围；大纲预览时提示篇幅

**[R5] SKILL.md 超 500 行** → 将项目分析逻辑放 `references/analysis-guide.md`、教学设计原则放 `references/pedagogy-guide.md`、模板放 `templates/`，SKILL.md 只做流程编排

**[R6] Mermaid 渲染兼容性** → 只使用 Mermaid 广泛支持的图表类型（flowchart, sequenceDiagram, classDiagram, stateDiagram, gitgraph）；复杂图表降级为绘图描述
